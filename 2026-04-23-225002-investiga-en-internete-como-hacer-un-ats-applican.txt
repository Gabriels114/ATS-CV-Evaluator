  ---
  Como funciona un ATS en la industria (2026)

  Proceso de evaluación

  Los ATS modernos hacen 3 cosas en orden:

  1. Parse — Extraen texto del PDF/DOCX y mapean a campos estructurados (nombre,
   experiencia, educación, skills)
  2. Match — Comparan el CV contra la JD usando keyword matching + semántica
  (NLP/embeddings)
  3. Score — Calculan un puntaje ponderado

  Pesos de scoring que usa la industria

  ┌───────────────────────────────────────────┬──────┐
  │                 Dimension                 │ Peso │
  ├───────────────────────────────────────────┼──────┤
  │ Hard skills match                         │ ~30% │
  ├───────────────────────────────────────────┼──────┤
  │ Experiencia relevante (con recencia bias) │ ~25% │
  ├───────────────────────────────────────────┼──────┤
  │ Educación                                 │ ~15% │
  ├───────────────────────────────────────────┼──────┤
  │ Título/seniority alignment                │ ~10% │
  ├───────────────────────────────────────────┼──────┤
  │ Logros cuantificados (métricas)           │ ~10% │
  ├───────────────────────────────────────────┼──────┤
  │ Soft skills                               │ ~5%  │
  ├───────────────────────────────────────────┼──────┤
  │ Formato/parseabilidad                     │ ~5%  │
  └───────────────────────────────────────────┴──────┘

  Técnicas NLP usadas

  - Keyword matching: exacto + fuzzy + semántico
  - NER (Named Entity Recognition): extrae skills, empresas, fechas
  - TF-IDF + BERT embeddings para similitud semántica
  - Recency bias: experiencia reciente pesa más

  Umbral de score típico

  - 80%+: Fuerte match, pasa filtros
  - 60-79%: Aceptable
  - 40-59%: En riesgo
  - <40%: Rechazado automáticamente

  ---
  Propuesta: Tu ATS personal

  Construyo un CLI en Python que:
  1. Lee tu CV (PDF o DOCX)
  2. Lee una job description (texto o URL)
  3. Usa Claude API para análisis semántico profundo
  4. Genera un reporte con score y feedback accionable

  Stack: Python + Claude API + pdfplumber + python-docx

  Output: Reporte en Markdown/JSON con score por dimensión, keywords faltantes,
  y sugerencias concretas para mejorar el CV.

  ---
  Antes de arrancar, dos preguntas:

  1. Formato de tu CV: ¿PDF, DOCX, o ambos?
  2. Interface: ¿CLI simple (comando en terminal) o prefieres una pequeña web
  UI?

⏺ Aquí lo que encontré comparando con la industria:

  ---
  Lo que hacen los ATS reales que nosotros no hacemos aún

  Lo que ya tenemos ✅

  - Recency bias (decay por años de antigüedad)
  - Keyword normalization con aliases
  - 7 dimensiones ponderadas
  - Extracción local sin API

  Brechas importantes

  1. Embeddings semánticos — el gap más grande
  Los ATS modernos (y el paper Resume2Vec) usan BERT/sentence-transformers para
  entender que "ML engineer" y "machine learning engineer" son lo mismo.
  Nosotros solo hacemos matching exacto + aliases manuales. Con
  sentence-transformers (100% local) ganaríamos +15% de precisión en el match de
   skills.

  2. Placement weighting — hoy no lo tenemos
  Skills mencionados en el título o sección "Skills" pesan 2–3x más que los
  mismos skills enterrados en un bullet point. Workday y Greenhouse aplican
  esto.

  3. Temporal decay de skills en el CV — parcialmente lo hacemos
  Marcamos experiencia antigua con recency bias, pero no penalizamos skills
  obsoletos (>7 años sin mencionar + no relevantes en el mercado actual).

  4. Taxonomía O*NET/ESCO — tenemos un JSON propio pero limitado
  O*NET tiene +35,000 ocupaciones y ~18,000 skills con relaciones jerárquicas.
  Integrarla mejoraría el matching de títulos y skills relacionados.

  5. Validación de contacto — no la hacemos
  Si falta email, teléfono o LinkedIn, muchos ATS descalifican automáticamente
  antes de leer el CV.

  ---
  Mi propuesta — ¿cuál implementamos primero?

  Dado que somos 100% local y sin API, lo más impactante con menor esfuerzo es:

  ┌───────────┬────────────────────────────────────────┬──────────┬─────────┐
